FairHire AI
Governance Heavy Project Report
GenAI application that is: compliant, bias-audited and meets the NYC Local Law 144, EEOC, and EU AI Act requirements.

Project Overview

Our FairHire AI is a generative AI-powered application screening tool designed to help various organizations automate the initial review of resumes while ensuring compliance with employment law and bias mitigation requirements. Our system design combines resume screening, bias detection, and audit logging, human-in-the-loop to create a transparent, auditable hiring process.

Key Innovation
FairHire AI uniquely integrates bias auditing, PII scrubbing, and explainable scoring directly into the application workflow. Unlike traditional ATS systems, every hiring decision is logged, bias is automatically detected, and human override is mandatory—ensuring humans remain in control.

🎯 Primary Goal

Accelerate candidate screening and provide a transparent tool for recruiters and managers to use.

⚖️ Compliance Focus

Meet NYC Local Law 144 (bias audit reporting), EU AI Act (high-risk system governance), and EEOC and Title VII employment law requirements.

🔍 Unique Control

Use PII scrubbing before Claude review, disparate impact ratio monitoring (EEOC 4/5ths rule), and audit logs for every action in the screening tool.

👥 Human-in-Loop

Human decides weather accept or override AI recommendation. Recruiters must certify that all AI recommendations were reviewed before sending resumes to hiring manager.

The Problem Statement
Our AI agent FairHire AI is designed to solve multiple problems in a recruiting environment.

Operational Challenge

FairHire Ai helps the business screen through hundreds of resumes quickly to pull minimum qualified candidates to review. The agent will review applicants' resumes for education, experience and minimum requirements against the advertised position and provide a match score. Recruiter then will have to go in and validate each applicant to either confirm or override the agent's output. Applicants who meet the minimum criteria will then be included on a pdf report which will be generated by agent so that it can be forwarded to the hiring manager for possible interview.

Compliance Challenge

The AI agent includes data testing to provide bias review and ensure compliance with NY 144 law, Equal Employment Opportunity Commission and compliance with EU AI Act Article 10. The bias testing can be completed in the application itself using administrator account. The administrator will also have access to review audit logs for last 120 days that can be exported as JSON.

Model & Infrastructure

AI Model: Claude 3 Haiku (claude-3-haiku-20240307) via Anthropic API
Justification: Fast and low-cost.

Data Inputs

  • Resumes: Recruiter uploaded PDFs, TXTs, and DOCX files.
  • Job Descriptions: Hiring Manager or Recruiter created. Stored in database with required skills, experience, education, and certifications
  • Test Datasets: Kaggle Resume Dataset.

Core Workflow

Resume Ingestion Pipeline:
File upload → Text extraction (pdf-parse) → PII scrubbing (Microsoft Presidio or 10 regex categories if Presidio fails) → Scrub validation → Database persistence

Screening Workflow:
Anonymized resume text + job criteria → Claude API → Structured JSON response → Validation for prohibited content → Score computation (skills 40pt, experience 35pt, education 15pt, certifications 10pt) → Results + prompt logged to database

AI Agents

Bias Detection Agent

Selection Rate Monitor

Audit Logging Agent

Compliance Export Agent

User Roles & Access Control

  • Administrator: Full access, disabling other user accounts, manager and recruiter view.
  • Recruiter: Uploading and screening resumes, overriding AI recommendation, creating job descriptions.
  • Manager: Creating job descriptions to include weighted scoring adjustment based on the vacancy requirements, making selections from certified list.

Human Review Points

  • Consent Gate: Recruiter must click "I confirm that I have obtained the candidate's consent to process their resume data. The data will be anonymized (PII removed) before AI screening and stored for a maximum of 120 days. Candidates can request deletion at any time." before resumes are uploaded to the FairHire AI.
  • Recruiter Override: Recruiter must review AI recommendation and accept or override the decision. Override requires a reason for the change.
  • Bias Alert Review: FairHire AI includes bias audit so that recruiter or administrator can stop the review.

Outputs & Reports

  • Hiring Manager report with recommended resumes
  • Applicant list with scores and explanations
  • Bias audit report (quarterly)
  • NYC Local Law 144 compliance exports (PDF + JSON)
  • Audit trail (immutable logs of all decisions)
Risk Assessment
Risk Framing
Stage-by-stage risk identification and mitigation

Design Stage Risks

⚠️ Regulatory Misalignment Risk

Misinterpreting requirements from NYC Local Law 144 and EU AI Act which could lead to compliance concerns.

Mitigation: Create an annual independent audit workflow for the application. Ensure independent testing on data which should include bias, PII scrub, compliance requirements are conducted and published.

⚠️ Biast Testing on historical data

Ensuring that bias concerns are addressed from the design stage.

Mitigation: Completing bias testing on the data verifying that the score is 0.8 or above and tunning the model.

Build Stage Risks

⚠️ Representative Data Risk

Training/Testing data from Kaggle failing to reflect current real-world market standards.

Mitigation: Reviewing test data and obtaining more current database of resumes.

⚠️ PII Leak

Accidental leak of the PII to the agent during the resume scoring or in JSON export.

Mitigation: Add Microsoft Presidio scrubbing layer with 10 regex categories before resume data is evaluated.

Validation Stage Risks

⚠️ Bias

The AI agent scores the protested groups (age/race/sex) lower than others and violates the EEOC standards.

Mitigation: Run Counterfactual Testing with the model: (swapping names) and the 4/5ths Rule threshold alert (DIR < 0.80).

Deployment Stage Risks

⚠️ Regulatory misalignment risk

Misinterpreting requirements from NYC Local Law 144 and EU AI Act which could lead to compliance concerns.

Mitigation: Create an annual independent audit workflow for the application. Ensure independent testing on data which should include bias, PII scrub, compliance requirements are conducted and published.

Operating Stage Risks

⚠️ Hallucinations

Claude inventing qualifications, skills or degrees that are not in resumes.

Mitigation: Immutable append-only audit logs. Daily backups. Monthly audit log review. Automated alerts for missing data.

⚠️ API Loop

The bias-detection agent gets stuck in infinite loops exceeding token usage.

Mitigation: Agent will be set at $5 limit for its usage with a dashboard that monitors the usage.

⚠️ Audit Failure

The audit logs are not created properly which would lead to non compliance with EU AI Act.

Mitigation: Application will run logs for a 120-day period, stored in Supabase.

Evolve Stage Risks

⚠️ Regulatory Risk/Legal Liability

Regulatory changes are not being reflected in the AEDT tool and application is no longer compliant. Incorrect scoring could expose penalties or bring lawsuits.

Mitigation: Ensure the compliance department communicates any regulatory changes to the technical team so that they can update how the system works, and set updated guardrails. Ensure independent audits are completed at least annually.

Key Risk Management Priorities

Bias Audit Data Integrity Remove Proxy Variables Secure PII Prompt Injection Prevention Data Drift Monitoring

Governance Framework

The system is classified as “High-Risk” due to:

  • NIST RMF Classification: High risk on civil rights, social welfare, it impacts human opportunity: Employment decisions - the agent provides recommendations to recruiters and hiring managers on which candidate is the best fit which impacts people's livelihoods, their career trajectory, and income
  • EU AI Act Classification Annex III if the system indirectly discriminates, Annex II (High Risk) even when using anti-discrimination guardrails. Ensuring fair treatment and non-discrimination ensures that protected groups (age, race, gender, disability, religion, national origin) rights are not violated.
  • Historical bias through training models on data that had historical bias. The model should be tested on a regular basis, and have at least one annual independent test to ensure compliance. Ensure testing includes protected groups.
  • Explainability any rejected applicant should be able to challenge the decision and receive a response on how the decision was made, which should be in plain english so that it is understandable.

Control Architecture and Policy-to-Control Translation

Our FairHire AI design is specifically designed to comply with the NYC Local Law 144, EU AS Act, and EEOC 4/5th rule (80%), the Title VII of the Civil Rights Act that prohibits hiring discrimination based on race, color, religion, sex, or national origin.

Compliance with New York 144 Law plan:

  • As a new AEDT Tool, and we have no historical data to test we decided to utilize Kaggle. Using a test data set to test for bias - we tested our model utilizing the Kaggle data set that included 2,481 applications in pdf format; the AI selected 707 resumes with overall selection rate at 28.5%. The initial test passed the gender test and failed in ethnicity. We used a bias mitigation strategy, using reweighting pre-processing technique which adjusts candidate weight so underrepresented groups receive equitable consideration without discarding any applications. This technique is used by IBM’s AIF 360 toolkit.
  • Our deployment plan also involves scheduling an independent bias audit to meet the annual requirement. At minimum our independent auditor would test and provide calculations of selection or scoring rates and the impact ratio across sex categories, race/ethnicity categories, and intersectional categories.
  • Our plan also involves publishing the testing summary of the most recent bias test that would include: the date of the most recent bias audit of the AEDT, The source and explanation of the data used to conduct the bias audit, The number of individuals the AEDT assessed that fall within an unknown category, The number of applicants or candidates, the selection or scoring rates, as applicable, and the impact ratios for all categories.
  • Our current version of the tool does not collect resumes directly from the applicant so to meet the automatic notice to the applicants that the employer uses AEDT tool our plan would be to request that the recruiter include notification of the tool used and receive consent from every applicant that their resume can be scanned. The recruiter certification will be required inside the user interface to confirm that the recruiter obtained a consent.

Compliance with EU AI Act plan

  • Our AI agent falls under the high risk category of the EU AI Act [2], which requires continuous risk management lifecycle. EU requires the AI tool to be registered in their database. Article 49: Registration | AI Act Service Desk
  • When designing the system we must take under consideration the Fundamental Rights Impact Assessment and evaluate how our AI agent could unfairly treat the applicants based on their age, gender or ethnicity. Another step of the design is to document how our agent will collect data and validate it. The last step in our risk management would be to include filters that would prevent bias.
  • The EA AI Act also specifies what technical requirements for the AI agent: Transparency and explainability - agent design must produce rational explanation why decision was made Logging and traceability - AI agent must include event log history of at least 6 month for audit purposes Accuracy and robustness - AI agent must be tested against the “golden dataset” of diverse resumes across different formats like word or pdf and different languages.
  • Other AI agent requirements we will include: Human in the loop - our guardrails will include human review before rejection notification is sent, allow for a recruiter to be able to override AI agents decision or ranking of candidates Training - all recruiters using the AI agent will be trained to recognize any signs of bias in the selection.
  • Notification requirements, our team will include notifications as per EU AI Act to include: Disclosure notice Right to explanation GDPR Alignment

Use: Initial training and bias testing. Anonymized resumes for synthetic demographic testing.

Data Quality Assurance

Resume Quality Checks

  • File format validation (PDF, DOCX, TXT)
  • Text extraction validation (ensure text is readable, not corrupted)
  • PII detection and removal (MS Presidio)
  • Length validation (must contain extractable text)

Job Description Quality Checks

  • Completeness: required skills, experience, education, certifications populated
  • Bias review: avoid coded language (e.g., "age", digital native", "team fit")

Scoring Validation

  • JSON schema validation on Claude response
  • Score range validation (0–100)
  • Weight verification (40+35+15+10 = 100%)
  • Spot check: manual review of 5–10 scored resumes weekly

Data Retention & Privacy

Data Type Retention Period Deletion Method
Raw resume files 7 years as per legal requirement Encrypted in S3; deleted via lifecycle policy
Extracted resume data (anonymized) 120 days post-hire/no-hire Database deletion
Audit logs 3 years (compliance) Deleted after 3 years
Bias audit reports Indefinite (regulatory requirement) Stored securely; PII references removed

Data Risks & Mitigations

🔴 Risk: Regulation Changes:

Impact and Mitigation: Regulation changes - any updates to regulations and policies need to be communicated to the design team and recruitment team to make sure if required changes are not implemented the tool can’t be used.


🔴 Risk: PII leakage:

Impact and Mitigation: engineering team should be conducting monitoring to ensure Presidio is working as expected.


🔴 Risk: Unauthorized Access:

Impact and Mitigation: the administrator has an access to disable other user accounts, as soon as there is a change in personnel administrator should disable the account. Also users should be educated that sharing of the account credential is not allowed and their password should be stored correctly.

Validation Framework

Unit Testing (Code Level)

  • PII scrubber: test 100+ regex patterns against known PII examples
  • Score computation: test weight calculation (40+35+15+10 = 100)
  • JSON parsing: test malformed responses, missing fields, type validation
  • Database queries: test audit log writes, immutability, query performance

Integration Testing (End-to-End)

  • Resume upload → text extraction → PII scrubbing → Claude API → scoring → database write
  • Override workflow: recruiter decision → audit log → email notification
  • Bias alert workflow: selection rates calculated → 4/5ths rule check → escalation to CHRO

Bias Testing (Model Fairness)

  • Disparate Impact Testing: Generate 6 synthetic resumes (identical skills, different demographic names). Compute selection rate ratio for each group. Pass if all ratios ≥ 0.80.
  • Fairness Metrics: Measure demographic parity, equalized odds, and predictive parity across gender, race, age, and disability proxies.
  • Edge Case Testing: Underrepresented groups, non-traditional career paths, gaps in resume, etc.

Security Testing

  • Penetration testing (authorized security scan)
  • API key/credential scanning (code review, secret detection)
  • PII leakage test (prompt injection with PII in resume text)
  • Access control validation (RBAC testing)

Compliance Testing

  • Audit log completeness: verify every action logged with timestamp and user ID
  • Data deletion verification: confirm GDPR "right to erasure" works end-to-end
  • Regulatory export validation: NYC 144 report format, required fields

Sign-Off Criteria

Component Test Requirement Sign-Off Authority
PII Scrubber False negative rate < 1% on 1000-sample validation set Data Science Lead
Bias Audit Disparate Impact Ratio ≥ 0.80 for all protected groups Responsible AI Lead + Legal
Scoring Algorithm 100% of scores fall within valid range (0–100); weights sum to 100 ML Engineering Lead
Security Scan Zero critical or high-severity vulnerabilities; no API keys exposed IT Security Lead
Audit Trail 100% of actions logged with timestamp, user ID, decision state Compliance Officer
Human-in-Loop All overrides documented; no batch auto-advance observed CHRO / QA Lead

Test Data Strategy

  • Kaggle Dataset: 24 job categories, 200K+ candidates for representative testing
  • Synthetic Resumes: Team-generated with controlled demographic variations for bias testing
  • Edge Cases: Career gaps, job-switching, multiple languages, industry transitions
  • Real Anonymized Data (Post-Launch): Validate on organization's historical hiring data

Continuous Validation (Post-Launch)

  • Daily: Monitor bias metrics dashboard. Alert on drift.
  • Weekly: Manual spot-check of 5–10 scored resumes.
  • Monthly: Review override patterns. Escalate if systematic bias detected.
  • Quarterly: Formal bias audit. Report to leadership and board.
  • Annually: Independent third-party audit. Public compliance report.
